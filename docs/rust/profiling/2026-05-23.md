# SPECTER Clone Profiling (2026-05-23)

Date: 2026-05-23

## Scope

Bounded profiling checked whether SPECTER/vector cloning is a current Rust
cleanup priority.

Primary artifact:

- `scratch/specter_clone_profile_20260523/summary.json`

## Results

JSON compatibility path, synthetic `from_json_paths`, 10,000 papers by 768
float32 dimensions, `preprocess=false`, three isolated runs:

| Workload | Total median | Vector/PaperData phase | Peak RSS delta |
|---|---:|---:|---:|
| No SPECTER | `0.221s` | `0.009s` | `22.7 MB` |
| NumPy dict SPECTER | `0.247s` | `0.040s` | `56.5 MB` |
| Pickle tuple SPECTER | `0.246s` | `0.038s` | `58.7 MB` |

With `preprocess=true`, one 10,000-paper comparison was `0.578s` without
SPECTER versus `0.624s` with NumPy SPECTER.

Direct Arrow fixture, `tests/fixtures/arrow/pubmed_specter2`, 1000 seeds, 100
queries, 25,000 candidate pairs:

| Workload | Total median | Peak RSS delta |
|---|---:|---:|
| Without SPECTER | `0.071s` | `15.8 MB` |
| With SPECTER | `0.131s` | `20.9 MB` |

Construction-related deltas were small on the direct Arrow fixture:
`read_specter` was about `6.7 ms`, summary construction was about `+4.6 ms`,
and feature assembly was about `+1.7 ms`.

## Suspected Copy Sites

- `s2and_rust/src/lib.rs` NumPy `slice.to_vec()`
- Python/list `Vec<f64>` to `Vec<f32>` conversion
- `from_json_paths` SPECTER extraction while building `PaperData`
- Per-vector norm computation
- Arrow summary exemplar `to_vec()` copies

## Decision

Do not change code for this now. The cost is measurable on the JSON
compatibility path, but it is not a clear current bottleneck on the
production-oriented Arrow path.

If this becomes important, rerun the same scratch commands with native
allocation profiling (`heaptrack`/`perf` on Linux or ETW allocation tracing on
Windows) to get stack-level attribution before editing clone sites.
